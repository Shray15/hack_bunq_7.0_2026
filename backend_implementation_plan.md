# Backend Implementation Plan

Companion to `backend_overview.md`. Splits the build into ordered phases so each phase ships something demoable on its own.

## Updated infrastructure decisions (supersede `backend_overview.md`)

- **Database:** Postgres 16 (Docker container on the same EC2), not SQLite.
- **Environment:** EC2 from day one. No local-only mode. The dev loop is `git push â†’ GitHub Actions â†’ EC2`.
- **EC2:** Ubuntu 24.04 LTS, `t3.small`, us-east-1, 30 GB gp3. SSH user `ubuntu`.
- **Networking:** No tunnel. EC2 security group opens SSH (your IP) + TCP 4567 (public). Backend binds `0.0.0.0:4567` and is hit directly.
- **LLM:** AWS Bedrock (Claude via `boto3`), not the Anthropic SDK directly. Model: `us.anthropic.claude-sonnet-4-5-20250929-v1:0`.
- **Registry:** GitHub Container Registry (GHCR).
- **Secrets:** GitHub Actions Secrets â†’ written to `/opt/app/.env` (mode 600) on EC2 at deploy time.
- **Migrations:** Alembic, autogenerated, applied on container start.
- **Deploy gating:**
  - `main` push â†’ automatic deploy
  - Any other branch â†’ manual deploy via `workflow_dispatch`

## Phase guide

Each phase has: **goal**, **tasks**, **exit criteria**, **dependencies for the next phase**. Phases 1â€“5 are sequential; later phases assume the previous phase shipped. Phase 0 is one-time infra and runs before everything else.

A phase is "done" when its exit criteria pass against the live EC2 deployment, not against a local laptop.

## Phase 2 â€” Recipe brain (Claude + Gemini + nutrition)

**Goal:** `/chat` and `/recipes/generate` produce real, validated recipes with real AI-generated images. Recipe library (history + favorites) works.

**Tasks**

1. **DB:** add `recipes` and `food_db_cache` tables; migration applied.
2. **Claude adapter (`adapters/claude.py`)**
   - Uses AWS Bedrock via `boto3` (`bedrock-runtime` client), not the Anthropic SDK directly.
   - Model: `AWS_BEDROCK_MODEL_ID` env var (default `us.anthropic.claude-sonnet-4-5-20250929-v1:0`).
   - System prompt with `cache_control: ephemeral` (recipe JSON schema, dietary rules, refusal handling).
   - Tools: `emit_recipe(...)` (forced), `emit_constraints(...)` (used by NLU step).
   - Helpers: `parse_transcript_to_constraints(transcript, profile)` and `generate_recipe(constraints, profile, today_macros)`.
3. **Gemini adapter (`adapters/gemini.py`)** â€” Nano-Banana image generation, takes `(name, key_ingredients) â†’ image_url` (uploaded to S3 or kept as data URL for demo).
4. **Nutrition adapter (`adapters/nutrition.py`)** â€” OpenFoodFacts query â†’ cached `food_db_cache` row â†’ per-100g macros. USDA fallback if OFF returns nothing usable.
5. **Recipe orchestration (`orchestrator/chat_flow.py`)**
   - `/chat` flow: NLU â†’ recipe gen â†’ persist â†’ fire `recipe_complete` SSE â†’ schedule image task â†’ schedule nutrition validation task.
   - `/recipes/generate` flow: same minus the NLU step.
6. **Background tasks (`background/`)**
   - `images.py` â€” Gemini call, update `recipes.image_url`, push `image_ready` SSE.
   - `nutrition.py` â€” for each ingredient, fetch+cache OFF, recompute totals, update `recipes.validated_macros`. Push `recipe_complete` only after this lands so iOS shows trustworthy numbers.
7. **Recipe library endpoints**
   - `GET /recipes/{id}`, `GET /recipes` (filters: favorited, since), `POST /recipes/{id}/favorite`, `POST /recipes/{id}/recook`.
8. **Per-user rate limit** on `/chat` and `/recipes/generate` (in-memory token bucket, 5 req/min default) to control LLM cost.

**Exit criteria**

- `/chat` with `transcript: "I'm hungry, want high-protein under 600 kcal"` returns within ~2 s, then SSE pushes `recipe_complete` (with validated macros) and `image_ready` (with a real Nano-Banana image) within ~15 s.
- Validated macros within 15% of LLM estimates on 5+ test recipes; mismatches logged.
- Favorite + recook round-trip works (favorite a recipe, recook creates a new draft cart referencing it).
- LLM cost per recipe under â‚¬0.05 with prompt caching (verify in AWS Cost Explorer / Bedrock usage).

## Phase 3 â€” Cart, MCP fan-out, bunq

**Goal:** From a generated recipe, produce a 3-store price comparison, allow line-item editing, mint a bunq payment URL, and detect "paid" via polling â€” all surfaced via SSE.

**Tasks**

1. **DB:** add `carts`, `cart_items`, `orders` tables; migration applied.
2. **MCP client (`adapters/mcp.py`)**
   - Connect to AH, Jumbo, Picnic MCP servers (HTTP transport, sibling containers on EC2).
   - Wrap tools `search_product`, `add_to_cart`, `get_total` per store.
   - `search_for_recipe(recipe, store)` returns `{items, total_eur, missing}`.
3. **Cart flow (`orchestrator/cart_flow.py`)**
   - `POST /cart/from-recipe` â€” parallel fan-out across 3 stores via `asyncio.gather`. Persist `carts` + `cart_items`. Push `cart_ready` SSE event with comparison.
   - `PATCH /cart/{cart_id}/items/{item_id}` â€” set `removed_at` or update `qty`. Recompute totals.
4. **bunq adapter (`adapters/bunq.py`)**
   - Sandbox auth handshake at startup.
   - `create_payment_request(amount_eur, description) â†’ {payment_url, request_id}`.
   - `get_payment_status(request_id) â†’ "pending"|"paid"|"declined"|"expired"`.
5. **Order flow (`orchestrator/order_flow.py`)**
   - `POST /cart/{cart_id}/checkout` with `{store}` â€” finalize cart selection, mint bunq URL, create `orders` row in `ready_to_pay`.
   - `GET /orders/{id}`, `GET /orders`.
   - State transitions emit `order_status` SSE.
6. **bunq status poller (`background/bunq_poll.py`)** â€” poll every 2 s for up to 5 min after `ready_to_pay`. On `paid`, transition state and push `payment_paid`. On timeout, transition to `payment_failed`.
7. **MCP order placement** â€” once `paid`, call MCP `place_order` (if available) and transition to `fulfilled` or `mcp_failed`.

**Exit criteria**

- End-to-end demo path (voice â†’ recipe â†’ cart with 3-store comparison â†’ checkout â†’ bunq URL â†’ paid status) works on a real iPhone against the live EC2 backend.
- `/cart/from-recipe` returns within 3 s for a 10-ingredient recipe.
- bunq sandbox payment is detected as `paid` within 5 s of completion in the bunq sandbox UI.
- Removing an item via `PATCH` updates the displayed total correctly.

## Phase 4 â€” Substitutions, meal logging, meal plan

**Goal:** The cart self-heals when stores miss ingredients, the calorie ring reflects real intake, and tomorrow's recipe is macro-aware.

**Tasks**

1. **DB:** add `meals_consumed` and `meal_plans` tables.
2. **Substitution flow**
   - When MCP returns `missing` items, fire async task that calls Claude `suggest_substitutions(ingredient, recipe_context) â†’ up to 3 alternatives`.
   - Retry MCP search per alternative in priority order; on first hit, replace the cart item silently and emit `substitution_proposed` (informational).
   - Cap retry depth at 1 substitution per ingredient.
3. **Meal logging endpoints**
   - `GET /meals/options` â€” recipes the user has either ordered (status >= `paid`) or marked `prepared` recently. Used by iOS as the "what did you eat" picker.
   - `POST /meals/log` â€” `{recipe_id, portion: 0.5}`; computes `calories_computed`/`macros_computed` from `recipes.validated_macros Ă— portion`; persists.
   - `GET /meals/today` â€” sum of today's logged macros + `remaining = profile_target - sum`.
   - `GET /meals/history?from=&to=`.
4. **Meal plan flow**
   - `POST /meal-plan/tomorrow` â€” gathers profile + today's logged macros + remaining weekly delta â†’ builds a constraint object â†’ calls `/recipes/generate` internally â†’ persists `meal_plans` row â†’ pushes `meal_plan_ready`.
   - `GET /meal-plan/upcoming` â€” returns next pending plan(s); schema supports a list so a weekly view is a UI-only extension later.
5. **Order â†’ ate mapping** â€” when an order goes `fulfilled`, optionally auto-create a `meals_consumed` row with `portion=1.0, source="order"`. iOS can offer to adjust the portion in the UI.

**Exit criteria**

- Demo recipe with one deliberately exotic ingredient (e.g. galangal) shows `substitution_proposed` and the cart still totals successfully.
- Logging two meals at portions 1.0 and 0.5 on `/meals/today` produces correct macro sums and remaining.
- `/meal-plan/tomorrow` produces a recipe whose macros + already-logged macros stay within ~10% of profile daily target on 3+ sample profiles.

## Phase 5 â€” Polish, resilience, observability

**Goal:** Production-ready behavior under all plausible demo paths, plus the operational hygiene to debug any failure live.

**Tasks**

1. **Resilience**
   - Retry policies (`tenacity`) on Claude, Gemini, OFF, MCP, bunq.
   - Per-adapter circuit breakers â€” after 3 consecutive failures, return a graceful error event on SSE instead of stalling the orchestrator.
   - bunq status poller cleanly cancels on order state change to avoid orphaned tasks.
2. **Observability**
   - `structlog` JSON logs to stdout; `docker compose logs -f backend` is the live view.
   - Add request id middleware; propagate to adapter logs.
   - `/metrics` Prometheus endpoint (optional â€” only if a Grafana panel adds demo value).
3. **Backups**
   - `pg_dump` cron container, ships compressed dump to S3 daily, retains 7 days.
   - Document restore procedure in `backend/docs/runbook.md`.
4. **Demo seeding**
   - `scripts/seed_demo.py` â€” creates demo user, sample profile, sample meal history, one favorited recipe, one fulfilled order. Idempotent.
   - GHA action on `main` runs seed only against a flag-protected env (do **not** auto-seed prod blindly).
5. **Performance tuning**
   - Confirm Anthropic prompt cache hit rate > 80% on the hot system prompt.
   - Pre-warm OFF cache for the top ~200 ingredients in advance.
   - Connection pooling on Postgres + httpx clients.
6. **Security review**
   - JWT secret rotated post-development.
   - Confirm `.env` on EC2 is mode 600 owned by docker user.
   - Cloudflare Tunnel access policy (Zero Trust) optionally locked to your team email for `/docs`.
7. **Backup demo path**
   - Recorded video of full happy path.
   - Local "offline mode" toggle in settings: when set, all adapters return canned data so a network outage at the venue doesn't kill the demo.

**Exit criteria**

- Cold-start of `docker compose up -d` to first successful `/chat` request completes in under 30 s.
- Killing any single adapter (Claude / Gemini / MCP / bunq) and replaying the demo path produces a graceful, user-readable error event â€” no hung SSE streams, no 500s.
- A clean `pg_dump` exists in S3 and a tested restore brings up an identical container.
- Logs in CloudWatch (or EC2-native if we skip CloudWatch) clearly trace one full demo request from `/chat` â†’ `recipe_complete` â†’ `image_ready` â†’ `cart_ready` â†’ `payment_paid` with consistent request id.

## Cross-phase conventions

- **Branching:** feature branches â†’ PR â†’ `main`. `development` (or any non-main branch) deploys only on `workflow_dispatch`. Deploys to `main` are auto, fast, and reversible by reverting the merge commit.
- **Migrations:** every schema change ships in the same PR as its model. CI runs `alembic upgrade head` on a throwaway Postgres.
- **Tests:** each adapter has a contract test with a recorded fixture (`vcrpy` or `respx` for HTTP, custom for MCP). Orchestrator flows have integration tests against the test Postgres.
- **Adapters before orchestrators:** never call an external service from a router; routers only call orchestrators, orchestrators call adapters.
- **SSE first, polling never:** any feature that involves "wait for X" must surface progress on `/events/stream`. iOS should never need to poll a backend endpoint.
